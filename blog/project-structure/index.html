Project structure

June 19, 2025 · Robert

Understanding Stattic's project structure is crucial for organizing your content, customizing your site, and maintaining a clean, scalable website. This guide explains every file and directory in a Stattic project and how they work together.

Overview

A typical Stattic project follows this structure:

my-website/
├── 📁 content/              # Your content (Markdown files)
│   ├── 📁 posts/           # Blog posts
│   ├── 📁 pages/           # Static pages
│   ├── 📄 authors.yml      # Author information
│   ├── 📄 categories.yml   # Category definitions
│   └── 📄 tags.yml         # Tag definitions
├── 📁 templates/           # Jinja2 templates (optional - falls back to built-in)
│   ├── 📄 base.html        # Main layout template
│   ├── 📄 index.html       # Homepage template
│   ├── 📄 post.html        # Blog post template
│   ├── 📄 page.html        # Static page template
│   ├── 📄 category.html    # Category archive template
│   ├── 📄 tag.html         # Tag archive template
│   └── 📄 404.html         # Error page template
├── 📁 assets/             # CSS, JS, images (copied to output)
│   ├── 📁 css/
│   ├── 📁 js/
│   └── 📁 images/
├── 📁 output/             # Generated site (after build)
└── 📄 stattic.yml         # Main configuration

Each directory serves a specific purpose in the site generation process.

Root Directory Structure

Essential Files

The heart of your Stattic project. Controls all site settings:

# Site information
site_url: https://example.com
site_title: My Static Site
site_tagline: Built with Stattic

output: output
content: content
templates: templates

  - Open Sans

## Content Directory Deep Dive

The `content/` directory is where all your site's content lives.

### Posts Directory (`content/posts/`)

Blog posts are stored here as Markdown files:

content/posts/ ├── 2024-01-15-my-first-post.md ├── 2024-01-20-learning-python.md ├── 2024-02-01-web-development-tips.md └── 2024-02-15-stattic-tutorial.md


**File Naming Best Practices:**
- Use date prefixes: `YYYY-MM-DD-title.md`
- Use hyphens instead of spaces
- Keep names descriptive but concise
- Be consistent across all posts

**Example Post Structure:**
```markdown
---
title: "My First Blog Post"
date: "2024-01-15"
categories: [1]  # Use numeric IDs from categories.yml
tags: [1, 2]     # Use numeric IDs from tags.yml
author: 1        # Use numeric ID from authors.yml
description: "My introduction to blogging with Stattic"
---

# My First Blog Post

Welcome to my blog! This is my first post...

Pages Directory (content/pages/)

Static pages that don't change frequently:

content/pages/
├── about.md
├── contact.md
├── projects.md
├── resume.md
└── privacy.md

Common Pages:

  • about.md - About you or your site
  • contact.md - Contact information
  • projects.md - Portfolio or project showcase
  • privacy.md - Privacy policy
  • terms.md - Terms of service

Example Page Structure:

---
title: "About Me"
description: "Learn more about me and my background"
custom_url: "about"  # Creates /about/ URL
---

# About Me

Hi there! I'm a web developer...

Templates System

The templates/ directory contains Jinja2 templates for customizing your site's appearance.

Default Template Structure

templates/
├── base.html           # Main layout template
├── index.html          # Homepage template
├── post.html           # Individual post template
├── page.html           # Static page template
├── category.html       # Category archive template
├── tag.html            # Tag archive template
└── 404.html           # Error page template

Template Hierarchy

Templates inherit from each other:

base.html (foundation)
├── index.html (homepage)
├── post.html (blog posts)
├── page.html (static pages)
├── category.html (category archives)
├── tag.html (tag archives)
└── 404.html (error page)

Custom Templates

You can override any default template by creating a file with the same name:

templates/
├── base.html           # Your custom base template
├── post.html           # Your custom post template
└── partials/           # Template fragments
    ├── header.html
    ├── footer.html
    ├── sidebar.html
    └── comments.html

Example Custom Base Template:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>{% block title %}{{ site_title }}{% endblock %}</title>
    <link rel="stylesheet" href="/assets/css/style.css">
</head>
<body>
    {% include 'partials/header.html' %}

    <main>
        {% block content %}{% endblock %}
    </main>

    {% include 'partials/footer.html' %}
</body>
</html>

Assets and Static Files

Assets Directory (assets/)

Files that are copied to the output directory:

assets/
├── css/
│   ├── stattic.css     # Main stylesheet (included by default)
│   ├── tailwind.min.css # Tailwind CSS (included by default)
│   └── custom.css      # Your custom styles
├── js/
│   ├── alpine.min.js   # Alpine.js (included by default)
│   └── main.js         # Your custom JavaScript
└── images/
    ├── logo.png
    ├── hero-bg.jpg
    └── icons/
        ├── github.svg
        └── twitter.svg

Note: Stattic automatically copies the entire assets/ directory to the output. It includes Tailwind CSS, Alpine.js, and a basic stylesheet by default when you run stattic --init.

Configuration Files

Main Configuration (stattic.yml)

Real Stattic configuration options:

# === SITE INFORMATION ===
site_title: "My Website"
site_tagline: "Thoughts, tutorials, and projects"
site_url: "https://example.com"

# === BUILD SETTINGS ===
output: "output"              # Output directory
content: "content"            # Content directory
templates: "templates"        # Templates directory (optional)
assets: "assets"              # Assets directory (optional)
blog_slug: "blog"             # URL slug for posts

# === CONTENT SETTINGS ===
posts_per_page: 10            # Posts per page for pagination
sort_by: "date"               # date, title, author, order

# === STYLING ===
fonts:                        # Google Fonts to include
  - "Inter"
  - "Source Code Pro"

# === SEO ===
robots: "public"              # public or private
llms: "public"                # public or private (for AI crawlers)

# === DEVELOPMENT ===
minify: true                  # Minify CSS and JS assets
watch: false                  # Watch mode (not implemented)

Supported Settings:

  • site_title, site_tagline, site_url - Site metadata
  • output, content, templates, assets - Directory paths
  • blog_slug - Custom URL slug for posts
  • posts_per_page - Pagination setting
  • sort_by - Post sorting (date, title, author, order)
  • fonts - Google Fonts list
  • robots, llms - SEO and AI crawler settings
  • minify - Asset minification
  • watch - File watching (not yet implemented)

Environment-Specific Configs

Since Stattic only looks for stattic.yml, stattic.yaml, or stattic.json in the current directory, you can manage different environments by:

  1. Copying different config files before building:

stattic.dev.yml (Development):

site_url: "http://localhost:8000"
minify: false
robots: "private"

stattic.prod.yml (Production):

site_url: "https://example.com"
minify: true
robots: "public"

Usage:

# Development build
cp stattic.dev.yml stattic.yml
stattic

# Production build  
cp stattic.prod.yml stattic.yml
stattic
  1. Using command line overrides:
# Override settings without changing config file
stattic --site-url https://example.com --minify --robots public

Best Practices

Organization

  1. Group Related Content

    content/posts/
├── tutorials/
│   ├── python-basics.md
│   └── web-development.md
├── reviews/
│   ├── book-reviews.md
│   └── tool-reviews.md
└── personal/
    ├── about-me.md
    └── my-journey.md

  2. Use Consistent Naming

    • Always use lowercase
    • Use hyphens instead of spaces
    • Be descriptive but concise

  3. Organize Assets by Purpose

    assets/images/
├── content/        # Content-specific images
├── ui/            # UI elements, icons
├── backgrounds/   # Background images
└── social/        # Social media images

Maintenance

  1. Keep Clean Structure

    • Remove unused files regularly
    • Organize content logically
    • Use meaningful directory names

  2. Version Control

    # Always ignore
output/
*.log
.DS_Store

# Consider ignoring
node_modules/
.venv/
__pycache__/

  3. Backup Strategy

    • Version control your source files
    • Regular backups of content
    • Document your structure

Performance

  1. Optimize Images

    • Use appropriate formats (WebP, AVIF)
    • Compress images before adding
    • Use descriptive alt text

  2. Minimize Assets

    • Enable minification in production
    • Remove unused CSS/JS
    • Optimize font loading

Troubleshooting Structure Issues

Common Problems

  1. Build Fails

    • Check file naming conventions
    • Verify YAML front matter syntax
    • Ensure required directories exist

  2. Missing Pages

    • Check file extensions (.md required)
    • Verify front matter is present
    • Check for typos in filenames

  3. Broken Links

    • Use relative paths consistently
    • Check case sensitivity
    • Verify target files exist

  4. Assets Not Loading

    • Check asset paths in templates
    • Verify files are in correct directories
    • Check for naming conflicts

Debugging Tools

# Check build process
stattic

# Get help and see all options
stattic --help

# Check version
stattic --version

# Analyze structure
find . -name "*.md" | head -20
ls -la content/
find output/ -type f | head -20

Summary

A well-organized Stattic project structure:

  • Makes content easy to find and manage
  • Scales as your site grows
  • Follows predictable conventions
  • Separates content from presentation
  • Enables efficient builds and deployment

Key Takeaways

  1. Content goes in content/ with clear organization
  2. Templates customize appearance and layout (optional - falls back to built-in)
  3. Assets are copied to output (CSS, JS, images)
  4. Configuration controls build behavior
  5. Consistent naming prevents problems
  6. Good organization saves time long-term

Start with the basic structure and expand as your site grows. Focus on consistency and clarity, and your Stattic site will be easy to maintain and scale.
